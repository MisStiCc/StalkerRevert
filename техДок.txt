ПОЛНАЯ ДОКУМЕНТАЦИЯ ПРОЕКТА STALKER REVERT
1. ОБЩАЯ АРХИТЕКТУРА
1.1 Основные принципы
SOLID-архитектура: каждый класс имеет единственную ответственность

Компонентный подход: сущности собираются из компонентов

Событийно-ориентированная: коммуникация через сигналы

Иерархия наследования: базовые классы → конкретные реализации

2. АВТОЗАГРУЗКИ (SINGLETONS)
2.1 GameManager
Назначение: Глобальное управление игрой
Ответственность:

Управление переходами между сценами

Сохранение и загрузка игры

Хранение глобальных данных (лаборатория, статистика)

Обработка результатов забегов

Взаимодействия:

Принимает команды от MainMenu и LabController

Передает параметры в ZoneController при старте забега

Управляет SaveData, LabData, GameStatistics

2.2 Signals
Назначение: Глобальная шина событий
Ответственность:

Проброс событий между несвязанными системами

Централизованная отправка уведомлений

Категории сигналов:

Игровые события (старт, пауза, конец игры)

События сущностей (спавн, смерть, взаимодействие)

События ресурсов (изменение энергии/биомассы)

События UI (открытие/закрытие панелей)

2.3 Enums
Назначение: Централизованное хранение перечислений
Ответственность:

Единое место определения всех типов

Типобезопасность в коде

Основные перечисления:

AnomalyType - типы аномалий

StalkerType - типы сталкеров

StalkerBehavior - модели поведения

MutantType - типы мутантов

Rarity - редкость артефактов

StalkerState - состояния сталкеров

TerrainType - типы местности

3. СИСТЕМА МЕНЕДЖЕРОВ
3.1 ZoneController (Главный оркестратор)
Назначение: Координация всех менеджеров во время забега
Расположение: zones/zone_controller.gd

Ответственность:

Создание и инициализация всех менеджеров

Проброс сигналов между менеджерами

Обработка запросов от HUD

Сбор результатов забега

Переход обратно в лабораторию

Взаимодействия:

Создает и содержит всех менеджеров как детей

Получает параметры забега от GameManager

Принимает сигналы от HUD (запросы аномалий/мутантов)

Передает результаты в GameManager

3.2 ResourceManager
Назначение: Управление ресурсами (энергия, биомасса)
Расположение: managers/resource_manager.gd

Параметры:

max_energy: максимальный запас энергии

max_biomass: максимальный запас биомассы

energy_regen_rate: скорость регенерации энергии

critical_threshold: порог критического уровня

Ответственность:

Учет текущего количества ресурсов

Регенерация энергии

Проверка возможности трат

Сигнализация о критических уровнях

3.3 SpawnManager
Назначение: Управление спавном сталкеров и мутантов
Расположение: managers/spawn_manager.gd

Параметры:

spawn_interval: интервал между волнами

min/max_stalkers_per_wave: количество сталкеров

spawn_radius: радиус спавна вокруг монолита

stalker_scenes: словарь сцен сталкеров

mutant_scenes: словарь сцен мутантов

Ответственность:

Генерация волн сталкеров

Выбор позиций для спавна (raycast)

Выбор типа сталкера по сложности

Спавн мутантов по запросу

Учет активных сущностей

3.4 AnomalyManager
Назначение: Создание и уничтожение аномалий и артефактов
Расположение: managers/anomaly_manager.gd

Параметры:

anomaly_scenes: словарь сцен аномалий

anomaly_artifact_map: связь аномалий с артефактами

artifact_values: ценности артефактов по редкости

difficulty_to_rarity: маппинг сложности на редкость

Ответственность:

Создание аномалий (проверка ресурсов)

Обработка уничтожения аномалий (создание артефактов)

Управление таймерами артефактов

Учет активных аномалий и артефактов

3.5 EventManager
Назначение: Управление глобальными событиями (выбросы)
Расположение: managers/event_manager.gd

Параметры:

pulse_duration: длительность выброса

pulses_to_win: количество выбросов для победы

difficulty_increase_per_pulse: рост сложности

Ответственность:

Запуск и завершение выбросов

Сброс артефактов у сталкеров во время выброса

Перемешивание аномалий

Проверка условий победы/поражения

3.6 ProgressionManager
Назначение: Управление прогрессией и сложностью
Расположение: managers/progression_manager.gd

Параметры:

base_difficulty: базовая сложность

difficulty_increase_per_run: рост за забег

pulses_to_win_base: базовое количество выбросов

health/speed/damage_scale_per_run: масштабирование

Ответственность:

Расчет текущей сложности

Учет статистики (убийства, создания)

Масштабирование характеристик сущностей

3.7 ParticleManager
Назначение: Визуальные эффекты частиц

Создание эффектов по требованию

3.8 SoundManager

Применение параметров типа

Управление таймерами (атака, обнаружение)

Обработка атак

Работа с артефактами (поднятие, бросание)

10. ВЗАИМОДЕЙСТВИЕ СИСТЕМ
10.1 Поток игры
Запуск → MainMenu

Новая игра → GameManager.start_new_game() → lab.tscn

Лаборатория → LabController отображает данные, ждет команды

Старт забега → GameManager.change_scene("run") → main.tscn

Забег → ZoneController управляет всем процессом

Завершение → ZoneController.finish_run() → GameManager.process_run_result() → lab.tscn

10.2 Создание сущностей
Сталкеры:

SpawnManager._start_wave() → _spawn_stalker()

Выбор позиции через raycast

Instantiate сцены, добавление в дерево

Регистрация в ZoneController

Мутанты:

HUD.mutant_requested.emit() → ZoneController._on_hud_mutant_requested()

Проверка ресурсов → SpawnManager.spawn_mutant()

Instantiate, добавление в дерево

Аномалии:

HUD.anomaly_requested.emit() → ZoneController._on_hud_anomaly_requested()

Проверка энергии → AnomalyManager.create_anomaly()

Instantiate, добавление в дерево

Артефакты:

Создаются при уничтожении аномалии

AnomalyManager.create_artifact()

Имеют таймер на превращение обратно

10.3 Управление ресурсами
Энергия:

Начальное значение: 500

Регенерация: +1 в секунду

Тратится на создание аномалий

Биомасса:

Начальное значение: 300

Пополняется: смерть сталкеров

Тратится на создание мутантов

Критический уровень (>80%) → выброс

10.4 Выброс (Radiation Pulse)
Достижение критической биомассы → EventManager.start_radiation_pulse()

Сброс биомассы до 30%

Остановка таймеров артефактов

Сталкеры сбрасывают артефакты

Аномалии перемешиваются

Через pulse_duration → завершение

Проверка на победу (pulse_count >= pulses_to_win)
